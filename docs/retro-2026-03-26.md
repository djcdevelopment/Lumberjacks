## What Worked Well

- **Incremental phases paid off.** Each phase built cleanly on the previous. Binary envelope (Phase 1) + InputQueue (Phase 2) + AoI filtering (Phase 3) meant Phase 4/5 just needed the serializer layer and transport тАФ no structural changes.
- **Optional UdpTransport constructor.** TickBroadcaster takes `UdpTransport?` тАФ existing JSON clients and unit tests unaffected.
- **Fast-path skip with `continue`.** Binary player_input in middleware calls `HandlePlayerInputBinary` and `continue`s the loop тАФ zero extra allocations on the hot path.
- **Crypto-random UDP tokens.** `RandomNumberGenerator.Fill` ensures tokens aren't guessable тАФ avoids session hijacking via UDP spoofing.

## What Was Awkward

- `session_started` originally used a typed `SessionStartedMessage` record. Adding `udp_token`/`udp_port` required switching to an anonymous object to avoid modifying the shared record тАФ slightly inconsistent. Could be cleaned up when a proper session handshake message is designed.
- No E2E test for UDP path yet. `test-multiplayer.js` validates the WebSocket path. UDP bind + dual-channel broadcast needs a new script.
