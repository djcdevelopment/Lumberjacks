Why this document exists

The next real capability is making the operation legible

The P7 deployment proved we can run an authoritative server. The next capability is not running a server — it is making that operation legible and trustworthy to the people whose community lives on it. This document is the single source of truth for what we are building, how it maps to what already exists, and how we grind it forward with minimal human interruption.

The one-line thesis. The network system already separates simulation authority, representation, delivery, and visibility. The dashboard adds a fifth cross-cutting concern the architecture does not yet name:

Accountability — people can see what happened, why the system believes it, and who had authority to change it.

Do not begin with Grafana as the product. Grafana / Cloud Monitoring is the engineering workbench underneath. The Lumberjacks dashboard is opinionated and community-readable, built around five concepts: health, safety, history, achievement, and authority.

02Three products in one interface

One interface answering three distinct questions for three distinct readers. The first vertical slice collapses these into four sections — Live / Safety / History / Trust.

Operator

Is the community safe & healthy right now?

Server health, players online, tick health, transport distribution, latency, gateway status. mostly LIVE

Community

What has this group accomplished together?

Achievements derived from authoritative observations, not badges stapled on. Every claim carries provenance. achievements shipped (D-07)

Stewardship

Can a nontechnical community operate this without blind trust?

Backups, changes, trust & roles, evidence, data-export authority. mostly GAP

04Claim provenance — reconciling two vocabularies

The repo already runs an evidence discipline and once did a naming-drift audit — so we must not fork the vocabulary. The dashboard’s four-tier achievement provenance is defined as an extension that maps onto the existing three-tier doc-provenance, on a distinct axis.

Existing — documentation provenance

From docs/current-focus.md. Answers “how do we know this claim about the system?”

  • Recorded — stated in a contemporaneous document/result
  • Observed — directly visible in current code or tests
  • Reconstructed — ordering inferred from technical dependencies

New — achievement provenance (dashboard)

Answers “how do we know this community accomplishment?” Same discipline, different axis.

  • Observed — directly derived from authoritative server events → maps to Observed
  • Verified — confirmed by multiple independent signals NEW
  • Community-awarded — explicitly granted by members/stewards; must be attributable NEW
  • Reconstructed — inferred from imported historical data → maps to Reconstructed

Rule: never present the four categories as equivalent. “Community-awarded” is the only tier backed by human authority rather than machine observation — which is exactly why it is the anchor for the Accountability concern: a community-awarded claim must always carry who awarded it and when, honoring the existing immutability rule (a correction is a new record, never an edit).

06Operating model — how we build this without you in the loop

Default posture: the agent drives, keeps the human out of the loop, and only pulls them in to prevent thrash. This section is the standing contract for that.

Delegation policy

  • Do it inline when the tool-use of batching/sending/retrieving exceeds the cost of just doing it.
  • Spawn a subagent for fan-out reads, isolated slices, and parallelizable build/verify work. Sonnet for mechanical, Opus for judgment-heavy slices.
  • Offload to HEARTH (local MCP door, :8710, ~35 tools, Ollama up) for lightweight generation and git-shaped chores — keep it warm, prefer it for cheap local work.

Pull-in triggers (interrupt the human)

  • A decision would fork the vocabulary or an invariant (lanes, tick budget, evidence tiers).
  • Cross-repo action touching C:\work\comfy or the disarmed P7 VM.
  • Anything irreversible, outward-facing, or in the safety-rule prohibited/permission list.
  • Ambiguity that would otherwise cause rework — surface it early to prevent thrash.

The build loop (per work item)

  1. Ground — confirm the signal’s real source against §03; never build on a metric that isn’t emitted.
  2. Slice — smallest shippable increment that adds one legible signal end-to-end.
  3. Delegate — pick inline / subagent / HEARTH per the policy above.
  4. Verify — exercise the actual surface (metric flowing, widget rendering), not just tests.
  5. Record — update this document (§03 grade + changelog) and tag evidence with provenance.

This document is the ledger. Every shipped increment flips a pill in §03 and adds a changelog row. A signal is not “done” until it renders in the community-readable dashboard and its source is provenance-tagged.

10Strategic context — the Community Compute platform

Three strategy docs (2026-07-11) situate this dashboard inside a larger arc: evolving Lumberjacks from a single authoritative server into a community-assisted distributed hosting platform — a cloud coordinator plus trusted volunteer nodes, with a control plane for node enrollment, capability reporting, placement, leases, snapshots, telemetry, and failover. This dashboard is that platform's legibility & accountability layer.

What we've built, mapped onto the platform build order

  • “Instrument tick phases & telemetry” → the Live slice + LumberjacksTelemetry, and as of 2026-07-12 per-tick-phase profiling (commit 6f1670d, branch claude/optimistic-mclean-435ef7): p50/p99/max per phase, DB-less via a “Tick timing” log window + GET /tick. First measurement attributes the 400-bot knee ~entirely to the broadcast send phase (152–168 ms p99 vs ~8 ms interest).
  • “Benchmark cloud vs AM4 under identical workloads” (step 5) → DONE (2026-07-12) — three-way sweep (cloud / AM4 / OMEN) + GPU-contention probe + off-host knee runs (docs/benchmark-host-capacity-2026-07-12.md). The core bet holds: local hardware carries the authority workload at sunk cost; co-tenancy with real local inference showed no RTT regression (“anytime”, not “off-hours-only”); and cloud and AM4 knee at the same 400 bots with CPU far from saturated — the wall is the single-threaded O(N²) broadcast fan-out, so capacity work now points at interest management (steps 3–4), not hardware.
  • Stewardship backlog reframed as the control plane: D-09 (identity/roles) = node enrollment & identity; D-08 (backups/restore) = snapshot management.
  • Evidence discipline already aligned: the Workbook's “evidence immutable, interpretation separate” is this repo's provenance model — Observed/Reconstructed events (evidence) vs. the achievement projection (interpretation). See §04.

SUT evolution (migration strategy)

The synthmon SUT is Stage 1 (VM + Compose, DB-less). Stages 2–4 add Cloud SQL, Caddy/TLS, Artifact Registry + CI/CD, and production monitoring — the concrete hardening path once behavioral parity holds.

Community capacity economy

Contributors earn non-transferable Hosting Credits for reliable compute (valued by availability, reliability, capability, network quality, regional demand), spent to launch private instances — a future Trust/Stewardship surface.

Guiding principle (Workbook): make existing capabilities reproducible, observable, and transferable before expanding scope — automate bookkeeping, not judgment.
